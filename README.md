# Thousand

**Thousand** — Android-приложение для просмотра онлайн-курсов с поддержкой избранного, локального кеширования и современной многомодульной архитектуры.

Проект создан как учебно-практическое приложение с упором на **чистую архитектуру**, **масштабируемость**, **тестируемость** и переход на **Jetpack Compose**.

---

## Возможности

- Просмотр списка курсов
- Загрузка данных с удалённого API
- Сохранение курсов в локальную базу
- Добавление/удаление курсов в избранное
- Экран избранных курсов
- Сортировка курсов по дате публикации
- Обработка состояний:
  - Loading
  - Content
  - Empty
  - Error
- Базовая подготовка под авторизацию
- Подготовка под onboarding flow

## Архитектура

Проект построен по принципам **Clean Architecture** с разделением ответственности между слоями:

### Модули
- `app` — точка входа, DI setup, Activity, application layer
- `domain` — бизнес-логика, use cases, repository contracts, domain models
- `data` — repository implementation, Room, Retrofit, data source
- `presentation` — UI, Compose screens, ViewModel, state/events/actions

---

## Архитектурные принципы

### `domain`
Слой `domain` **не знает**:
- о Room
- о Retrofit
- о Compose
- о Navigation
- о конкретной реализации репозиториев

Он содержит только:
- use cases
- интерфейсы репозиториев
- domain models

Это позволяет:
- легко тестировать бизнес-логику
- заменять реализацию data layer
- избегать жёсткой связанности

---

### `data`
Слой `data` отвечает за:
- работу с API
- локальное хранение данных
- маппинг data ↔ domain
- реализацию repository interface

---

### `presentation`
Слой `presentation` отвечает за:
- отображение UI
- обработку пользовательских действий
- управление экранным состоянием
- одноразовые события (navigation, snackbar)

Используется разделение на:

- **UiState** — постоянное состояние экрана
- **Action** — действия пользователя
- **Event** — одноразовые события

Пример:
- `Loading / Content / Empty / Error` → `UiState`
- нажатие на кнопку → `Action`
- показать snackbar / навигация → `Event`

## Управление состоянием

В проекте используется подход, близкий к **MVI-style state handling** внутри `ViewModel`.

### Почему так
Это даёт:
- предсказуемое поведение UI
- единый источник истины
- удобство отладки
- устойчивость к конфигурационным изменениям

### Разделение состояний
Пример типичных экранных состояний:
- `Loading`
- `Content`
- `Empty`
- `Error`

### Почему `Snackbar` не хранится в `UiState`
Потому что snackbar — это **одноразовое событие**, а не постоянное состояние экрана.

Если хранить snackbar в `UiState`, он может повторно отображаться после:
- поворота экрана
- пересоздания Activity
- повторного подписывания UI

Поэтому одноразовые действия вынесены в `SharedFlow<Event>`.

## Работа с данными

Приложение использует подход:

### Remote → Local → UI

Поток работы:
1. Данные запрашиваются с API
2. Сохраняются в локальную базу Room
3. UI подписывается на локальные данные через `Flow`

### Почему это хорошо
Такой подход даёт:
- оффлайн-доступ к уже загруженным данным
- стабильное отображение UI
- меньше зависимости UI от состояния сети
- более предсказуемую архитектуру

---

## Обработка ошибок

В проекте учитываются разные пользовательские сценарии:

### Если интернет недоступен и данных ещё нет
Показывается полноценный **Error screen**

### Если интернет недоступен, но локальные данные уже есть
Экран **не ломается**, а пользователь получает:
- старые сохранённые данные
- уведомление об ошибке через snackbar

Это поведение ближе к production UX, чем полное падение экрана при любом сетевом сбое.
